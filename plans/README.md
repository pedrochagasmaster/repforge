# RepForge improvement plans

Generated by an `/improve deep` pass that transforms
[`docs/persona-product-feedback-report.md`](../docs/persona-product-feedback-report.md)
into self-contained, executable implementation plans, complemented with a
codebase audit of `app.js`, `index.html`, `styles.css`, and `sw.js`.

- **Written against**: commit `ff67850` (`ff67850…`), 2026-07-01 (wave 2 direction
  pass). Wave 1 plans (001–016) were written at `1d68b68` and are **DONE** in
  the codebase (merged PR #13).
- **Codebase shape**: static PWA, no build step, no framework. Primary state in
  IndexedDB (`repforge` / `kv`) with `localStorage` mirror (`repforge_v1`), draft
  key (`repforge_draft_v1`). Logic in `app.js` (~692 lines), markup in
  `index.html`, styles in `styles.css`, offline caching in `sw.js` (`repforge-v7`).
- **Verification gate (every plan uses it)**: `node --check app.js` for syntax,
  then the Playwright simulation:

  ```bash
  # terminal 1 — static server from repo root
  python3 -m http.server 8000
  # terminal 2 — from repo root
  cd test && npm install && npx playwright install chromium && node simulation.mjs
  # expect: "FAILED: 0" and exit code 0
  ```

  There is **no** linter, type-checker, or bundler in this repo — do not look
  for one. `test/simulation.mjs` is the whole test surface (~80 checks at
  `ff67850`); executors add checks there for new behavior.

Every plan is written for the weakest plausible executor: exact `file:line`
current-state excerpts, repo conventions inlined, a drift check, verification
gates, hard scope boundaries, and STOP conditions. Plans never assume you saw
this index or the report.

## How to read a plan

Each file states its priority, effort, risk, category, dependencies, and the
persona(s) or audit finding it came from. Pick one, hand it to an executor
("implement `plans/001-*.md`"), and review the diff against its Done criteria.

## Product guardrails (from the report synthesis, treat as non-negotiable)

These constrain **every** plan. An executor that finds itself building any of
these should hit a STOP condition:

- No accounts, no cloud sync, no backend. Local-first is the product.
- No social feed, leaderboards, sharing, or gamification (streaks/badges/XP).
- No AI chat coach or auto-generated programs.
- No sixth nav tab; new depth ships hidden/collapsed/off the Log tab.
- Protect the Log tab's speed above all else.

## Explicitly rejected (do not re-file)

Consensus across ≥5 personas that these betray the product:

- Social feed / leaderboards / PR sharing / community.
- AI coaching chat, auto-generated programs.
- Full coach SaaS (billing, scheduling, nutrition, in-app messaging).
- Wearable / HRV / sleep / readiness dashboards.
- Exercise GIF/video library, form-check empire.
- Gamification: streaks, badges, XP, "forge level".
- Mandatory accounts / "RepForge Cloud" as the default story.
- A marketplace of programs. A sixth nav tab.

## Findings considered and rejected (wave 2 direction pass)

- **Social / AI / cloud / gamification**: rejected by product guardrails and ≥5
  personas — do not re-file.
- **Sixth nav tab** for any wave 2 feature: rejected; depth stays on Log or in
  collapsed Stats (`#statsDeep`).
- **Mesocycle blocks in wave 2**: real value but L-effort data model; deferred to
  backlog pending dedicated spike (not bundled into 017–021).
- **Flexible schedule now**: parent demand is real, but blast radius on day tabs
  exceeds wave 2 scope; trim + substitute address immediate pain.
